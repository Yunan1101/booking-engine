# Booking Engine 아키텍트 노트 (블로그 작성용)

> 이 문서는 향후 기술 블로그 작성 및 포트폴리오 정리를 위한 핵심 요약본입니다. 노션(Notion)에 복사해서 붙여넣기만 하면 예쁘게 포맷팅됩니다.

---

## 1. 프로젝트 초기 환경 세팅 및 도메인 모델링
* **작업 일자:** 2026-05-07
* **관련 커밋:** `feat: 콘서트 티켓팅 시스템 도메인 엔티티 설계 및 DB 연동`

### 왜 이렇게 설계했는가? (Why & Architecture Point)

1. **대규모 트래픽 대비 커넥션 풀(Connection Pool) 튜닝**
   * **문제 인식:** 수만 명이 동시에 몰리는 콘서트 티켓팅의 특성상, 기본 DB 연결 설정으로는 병목 현상이 발생할 수 있음.
   * **해결 방안:** Spring Boot의 기본 커넥션 풀인 `HikariCP`의 `maximum-pool-size`를 20으로, 타임아웃 설정을 튜닝하여 DB 연결의 안정성을 확보함.

2. **비관적 락(Pessimistic Lock)을 고려한 `Seat` 엔티티 설계**
   * **문제 인식:** 여러 명의 유저가 동시에 같은 좌석(예: VIP석)을 클릭할 때 발생하는 '초과 예매(Overbooking)' 동시성 문제.
   * **해결 방안:** `Seat` 엔티티를 독립적으로 설계하고, 이후 비즈니스 로직 작성 시 DB 레벨에서 X-Lock(`FOR UPDATE`)을 거는 비관적 락을 사용할 수 있도록 상태값(`AVAILABLE`, `RESERVED`)을 명확히 분리함.

3. **조회 성능 최적화 (복합 인덱스)**
   * **해결 방안:** 특정 회차(`concert_session_id`)의 예약 가능한 좌석(`status`)을 매우 빠르게 찾아오기 위해, `Seat` 테이블에 두 컬럼을 묶은 **복합 인덱스(Composite Index)**를 적용함. 이는 단순 CRUD를 넘어 대용량 데이터 조회를 고려한 설계임.

## 2. 핵심 비즈니스 로직(API) 구현 및 동시성 제어 적용
* **작업 일자:** 2026-05-07
* **관련 커밋:** `feat: 콘서트 예매 비즈니스 로직 및 비관적 락(Pessimistic Lock) 적용`

### 왜 이렇게 설계했는가? (Why & Architecture Point)

1. **JPA `@Lock`을 활용한 예매 동시성 제어**
   * **구현 방식:** `SeatRepository`에 `@Lock(LockModeType.PESSIMISTIC_WRITE)`를 선언하여 `findByIdForUpdate` 쿼리를 작성.
   * **선택 이유 (Why):** 낙관적 락(Optimistic Lock)은 충돌이 잦을 경우 재시도 로직을 구현해야 하고 DB 부하가 커질 수 있음. 콘서트 티켓팅처럼 **"특정 좌석에 대한 충돌(경쟁)이 100% 확실시되는"** 상황에서는 최초에 락을 강하게 걸어버리는 비관적 락이 데이터 정합성을 가장 확실하고 빠르게 보장함.

2. **글로벌 예외 처리기 (`@RestControllerAdvice`) 도입**
   * **구현 방식:** 누군가 이미 선점한 좌석을 요청했을 때 발생하는 `SeatAlreadyReservedException`을 전역에서 낚아채서 일관된 에러 응답(`ErrorResponse`)으로 내려주도록 구현.
   * **선택 이유 (Why):** 에러 처리 코드가 비즈니스 로직(Service)과 컨트롤러(Controller)에 파편화되는 것을 막고, 프론트엔드 개발자에게 규격화된 포맷(errorCode, errorMessage)을 제공하여 협업 효율성을 극대화하기 위함.

## 3. 기초 데이터(Dummy Data) 주입 및 동시성 제어 부하 테스트
* **작업 일자:** 2026-05-07
* **관련 커밋:** `test: 더미 데이터 주입 로직 추가 및 동시성 예매 부하 테스트 검증`

### 왜 이렇게 설계했는가? (Why & Architecture Point)

1. **자동화된 DataInitializer 세팅**
   * **구현 방식:** `CommandLineRunner`를 활용하여 서버 구동 시 초기 데이터(유저, 콘서트, 좌석 50개)를 자동 삽입.
   * **선택 이유 (Why):** 반복적인 수동 쿼리 작업을 줄여 개발 생산성을 높이고, 테스트 환경을 일관되게 유지하기 위함.

2. **쉘 스크립트를 이용한 극한의 동시 접속 테스트**
   * **구현 방식:** 쉘 스크립트의 백그라운드 실행(`&`)과 `wait` 명령어를 활용하여, 순식간에 20~100건의 비동기 POST API 요청을 서버로 발송.
   * **테스트 결과 및 의의:** 한 좌석에 다수가 동시 접근 시, 우리가 설정한 **비관적 락(Pessimistic Lock)** 메커니즘에 의해 단 1명만 `200 OK`(예매 성공)를 받고, 나머지는 모두 `409 Conflict`(초과 예매 차단)를 반환함을 직접 검증함.

## 4. API 문서 자동화 (Swagger UI 도입)
* **작업 일자:** 2026-05-07
* **관련 커밋:** `docs: Swagger UI 의존성 추가 및 API 명세서 자동화 적용`

### 왜 이렇게 설계했는가? (Why & Architecture Point)

1. **`springdoc-openapi`를 활용한 명세 자동화**
   * **문제 인식:** 백엔드 코드가 변경될 때마다 수동으로 API 문서를 수정하면 누락과 오류가 빈번히 발생하며, 프론트엔드 개발자와의 소통 비용이 급증함.
   * **해결 방안:** Spring Boot 코드 자체에 `@Operation`, `@ApiResponse` 등의 어노테이션을 부착하여, 코드가 곧 문서가 되도록(Docs as Code) 설정함. 서버 구동 시 실시간으로 최신 명세서가 생성되어 소통 비용을 획기적으로 낮춤.
